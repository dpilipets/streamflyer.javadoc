054     * <h3 id="g1">1. When shall I use this library?</h3>
055     * <p>
056     * This library provides filter that modify character streams on-the-fly with a
057     * small memory foot-print using the Java interfaces {@link Reader} and
058     * {@link Writer}. Additionally, you can use these filters to report some
059     * patterns found in character streams without modifying the content of the
060     * character streams.
061     * <p>
062     * Use this library only if you use Java 6 or higher.
063     * <h3 id="g3">2. How do I modify character streams?</h3>
064     * <p>
065     * First you have to implement a callback that makes the desired modifications
066     * on a small part of the entire character stream. The contract of that callback
067     * is defined by {@link Modifier}. For some common use cases pre-defined
068     * modifiers exist.
069     * <p>
070     * Second, wrap the original reader or writer with a modifying reader or writer.
071     * <p>
072     * Example for a modifying reader that removes invalid characters from XML
073     * files:
074     * <code><pre class="prettyprint lang-java">// choose the character stream to modify
075    Reader originalReader = ... // this reader is connected to the original data source
076    
077    // select the modifier
078    Modifier myModifier = new InvalidXmlCharacterModifier(1024, "", "1.0", false);
079    
080    // create the modifying reader that wraps the original reader
081    Reader modifyingReader = new ModifyingReader(originalReader, myModifier);
082    
083    ... // use the modifying reader instead of the original reader</pre></code>
084     * <h3 id="g4">3. Which modifiers are provided out-of-the-box?</h3>
085     * <p>
086     * The following modifications are provided out-of-the-box.
087     * <ul>
088     * <li>{@link RegexModifier}: Replaces characters using regular expressions.
089     * <li>{@link XmlVersionModifier}: Changes the version in the prolog of XML
090     * files.
091     * <li>{@link InvalidXmlCharacterModifier}: Replaces invalid XML characters in
092     * XML files or removes them from XML files.
093     * <li>
094     * {@link com.googlecode.streamflyer.experimental.range.RangeFilterModifier
095     * RangeFilterModifier} (experimental module): Removes everything from the
096     * stream apart from the content between a start tag and an end tag.
097     * </ul>
098     * The factories {@link ModifyingReaderFactory} and
099     * {@link ModifyingWriterFactory} ease the use of these modifiers.
100     * <p>
101     * Do you know another common use case that should be supported by this library?
102     * Then, please let the author know. A corresponding modifier might be added to
103     * the library in a future version.
104     * <h3 id="g5">4. What is the benefit of this library in comparison to sed or
105     * awk or Perl?</h3>
106     * <p>
107     * The line-based approach of <em>sed</em> consumes a lot of memory if you
108     * modify a large XML file that does contain no or only a few line breaks. The
109     * modifications <em>sed</em> can carry out are limited to replacements using
110     * regular expressions. This library instead supports any kind of modification
111     * and consumes not much memory how big the processed file may be. In comparison
112     * to <em>awk</em> and <em>perl</em>, this library seamlessly integrates with
113     * Java software that uses the interfaces for character streams ({@link Reader}
114     * and {@link Writer}).
115     * <h3 id="g7">5. Can I use regular expressions to modify streams?</h3>
116     * <p>
117     * Yes. Use the {@link RegexModifier} directly or, more convenient, use the
118     * {@link ModifyingReaderFactory}.
119     * <h3 id="g8">6. How much memory does a modifying reader or writer consume?</h3>
120     * <p>
121     * This depends entirely on the used {@link Modifier} and the task the modifier
122     * has to fulfill. Basically the modifier defines the number of characters to
123     * process at once by {@link AfterModification#getNewNumberOfChars()}. This is
124     * the minimum of characters in the memory but in many cases the maximum as
125     * well. The API documentation of each modifier implementation should make clear
126     * how much memory the modifier consumes.
127     * <h3 id="g9">7. How big is the performance overhead of an idle modifying
128     * reader or modifying writer?</h3>
129     * <p>
130     * If you use an idle modifier to process 10 million characters using a buffer
131     * of 500 characters, then the overhead is 0.2 seconds using a modifying reader
132     * and 0.4 seconds using a modifying writer (normal PC, 2011).
133     * <p>
134     * <h3 id="g10">8. How do I process regex patterns found in the character
135     * stream?</h3>
136     * <p>
137     * By default {@link RegexModifier} will replace the matches with the given
138     * replacement. If you want to do more or something else (logging, arbitrary
139     * insertions), then implement your own {@link MatchProcessor}, and use it with
140     * the {@link RegexModifier}.
141     * <p>
142     * <h3 id="g11">9. Is there a way to modify byte streams instead of character
143     * streams, i.e. is there something like a modifying input stream or output
144     * stream?</h3>
145     * <p>
146     * No, at the moment there is no special code for byte streams. An approach
147     * could be to define your own {@link CharsetDecoder} and {@link CharsetEncoder}
148     * , then convert the byte stream to a pseudo character stream, modify the
149     * character stream, and then convert the pseudo characters back to bytes. Is
150     * the performance of this approach still sufficient despite the two
151     * conversions? Let me know if you tried this.
152     *
