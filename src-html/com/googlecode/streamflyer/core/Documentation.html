055     * <h3 id="g1">1. When shall I use this library?</h3>
056     * <p>
057     * This library provides filter that modify character streams on-the-fly with a
058     * small memory foot-print using the Java interfaces {@link Reader} and
059     * {@link Writer}. Additionally, you can use these filters to report some
060     * patterns found in character streams without modifying the content of the
061     * character streams.
062     * <p>
063     * Use this library only if you use Java 6 or higher.
064     * <p>
065     * If you want to optimize the speed of the stream modifications and you don't
066     * care of memory consumption (assuming you are in the cloud), then load the
067     * entire content of the stream into the memory and modify the stream content
068     * there. In this case this library is not the right choice for you because this
069     * library minimizes the use of memory at the expense of a reduced speed of the
070     * stream modifications.
071     * <h3 id="g3">2. How do I modify character streams?</h3>
072     * <p>
073     * First you have to implement a callback that makes the desired modifications
074     * on a small part of the entire character stream. The contract of that callback
075     * is defined by {@link Modifier}. For some common use cases pre-defined
076     * modifiers exist.
077     * <p>
078     * Second, wrap the original reader or writer with a modifying reader or writer.
079     * <p>
080     * Example for a modifying reader that removes invalid characters from XML
081     * files:
082     * <code><pre class="prettyprint lang-java">// choose the character stream to modify
083    Reader originalReader = ... // this reader is connected to the original data source
084    
085    // select the modifier
086    Modifier myModifier = new InvalidXmlCharacterModifier(1024, "", "1.0", false);
087    
088    // create the modifying reader that wraps the original reader
089    Reader modifyingReader = new ModifyingReader(originalReader, myModifier);
090    
091    ... // use the modifying reader instead of the original reader</pre></code>
092     * <h3 id="g4">3. Which modifiers are provided out-of-the-box?</h3>
093     * <p>
094     * The following modifications are provided out-of-the-box.
095     * <ul>
096     * <li>{@link RegexModifier}: Replaces characters using regular expressions.
097     * <li>{@link XmlVersionModifier}: Changes the version in the prolog of XML
098     * files.
099     * <li>{@link InvalidXmlCharacterModifier}: Replaces invalid XML characters in
100     * XML files or removes them from XML files.
101     * <li>
102     * {@link com.googlecode.streamflyer.experimental.range.RangeFilterModifier
103     * RangeFilterModifier} (experimental module): Removes everything from the
104     * stream apart from the content between a start tag and an end tag.
105     * </ul>
106     * The factories {@link ModifyingReaderFactory} and
107     * {@link ModifyingWriterFactory} ease the use of these modifiers.
108     * <p>
109     * Do you know another common use case that should be supported by this library?
110     * Then, please let the author know. A corresponding modifier might be added to
111     * the library in a future version.
112     * <h3 id="g5">4. What is the benefit of this library in comparison to sed or
113     * awk or Perl?</h3>
114     * <p>
115     * The line-based approach of <em>sed</em> consumes a lot of memory if you
116     * modify a large XML file that does contain no or only a few line breaks. The
117     * modifications <em>sed</em> can carry out are limited to replacements using
118     * regular expressions. This library instead supports any kind of modification
119     * and consumes not much memory how big the processed file may be. In comparison
120     * to <em>awk</em> and <em>perl</em>, this library seamlessly integrates with
121     * Java software that uses the interfaces for character streams ({@link Reader}
122     * and {@link Writer}).
123     * <h3 id="g7">5. Can I use regular expressions to modify streams?</h3>
124     * <p>
125     * Yes. Use the {@link RegexModifier} directly or, more convenient, use the
126     * {@link ModifyingReaderFactory} or the {@link ModifyingWriterFactory}.
127     * <h3 id="g8">6. How much memory does a modifying reader or writer consume?</h3>
128     * <p>
129     * This depends entirely on the used {@link Modifier} and the task the modifier
130     * has to fulfill. Basically the modifier defines the number of characters to
131     * process at once by {@link AfterModification#getNewNumberOfChars()}. This is
132     * the minimum of characters in the memory. The API documentation of each
133     * modifier implementation should make clear how much memory the modifier
134     * consumes at most. Double that number (the maximum number of characters to be
135     * processed at once), then you got the maximum capacity of the used character
136     * buffer, i.e. the number of characters in the memory. Hint: The number must be
137     * doubled due to the internal implementation of Java's
138     * {@link StringBuilder#ensureCapacity(int)}.
139     * <h3 id="g9">7. How big is the performance overhead of a modifying reader or
140     * modifying writer that does nothing (no modification, no logging etc.) in
141     * comparison to the use of a reader or writer that reads its data directly from
142     * memory?</h3>
143     * <p>
144     * If you use such an idle modifier to process 10 million characters using a
145     * buffer of 500 characters, then the overhead is 0.2 seconds using a modifying
146     * reader and 0.4 seconds using a modifying writer (normal PC, 2011).
147     * <p>
148     * <h3 id="g10">8. How do I process regex patterns found in the character
149     * stream?</h3>
150     * <p>
151     * By default {@link RegexModifier} will replace the matches with the given
152     * replacement. If you want to do more or something else (logging, arbitrary
153     * insertions), then implement your own {@link MatchProcessor}, and use it with
154     * the {@link RegexModifier}.
155     * <p>
156     * <h3 id="g11">9. Is there a way to modify byte streams instead of character
157     * streams, i.e. is there something like a modifying input stream or output
158     * stream?</h3>
159     * <p>
160     * No, at the moment there is no special code for byte streams. An approach
161     * could be to define your own {@link CharsetDecoder} and {@link CharsetEncoder}
162     * , then convert the byte stream to a pseudo character stream, modify the
163     * character stream, and then convert the pseudo characters back to bytes. Is
164     * the performance of this approach still sufficient despite the two
165     * conversions? Let me know if you tried this.
166     *

168     * @since 13.09.2011
